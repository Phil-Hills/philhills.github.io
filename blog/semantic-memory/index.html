Semantic Memory for Agents

By Phil Hills · January 2, 2025

Agents that can’t remember behave like goldfish: clever in the moment, useless across workflows. If you want multi-agent systems to do real work, they need structured memory — not just bigger prompts.

Context Windows Aren’t Memory

A context window is an inbox, not a memory. Once the window moves on, the system has no durable representation of what happened. Every step has to be re-specified from scratch.

Real memory needs to be:

  • Persistent across steps and tasks
  • Structured so agents can navigate it
  • Semantic instead of just raw text
  • Portable across agents and tools

Why Semantic Memory, Not Just Logs

Logs are a record. Semantic memory is a substrate. The difference is structure and intent:

  • Logs: append-only traces, rarely reused directly
  • Semantic memory: curated, compressed, labeled state for future reasoning

Agents need the latter if you want them to behave like systems instead of one-shot chatbots.

Memory as Structured Packets

The foundation of my approach is to treat memory as structured packets. Cube Protocol is one such format: a reversible, labeled 3D “cube” that encodes both content and semantics.

Each packet tells an agent:

  • What domain it belongs to
  • How it was produced
  • What outcomes it relates to

What This Enables

Once memory is structured, agents can:

  • Preserve task state across long workflows
  • Hand off context between specialized agents
  • Perform targeted retrieval instead of reloading everything
  • Explain decisions based on prior state packets

From “Chat” to Systems

Giving agents semantic memory is the difference between “a clever model” and “an evolving system”. Without memory, your architecture is just a sequence of prompts. With memory, you have genuine state, continuity, and the ability to improve over time.
